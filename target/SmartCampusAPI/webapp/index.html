Smart Campus API Testing Guide

HTML test reference for the Smart Campus Sensor & Room Management API.

Project Overview

This Smart Campus API is running on Apache Tomcat 10.1 and is accessible through the base URL shown below. The system was developed as part of the 5COSC022W Client-Server Architectures coursework using JAX-RS / Jersey 3.1.3.

The API simulates a campus environment where:

  • Rooms can be created and managed
  • Sensors can be assigned to rooms
  • Sensor readings can be recorded and retrieved

All data is stored in memory (HashMap / ArrayList) as required by the coursework, so data will reset when the server restarts.

This API follows REST principles including:

  • Resource-based endpoints
  • Proper HTTP methods (GET, POST, DELETE)
  • Structured JSON error responses via custom exception mappers
  • Error handling with correct status codes (400, 403, 404, 409, 422, 500)
  • Request and response logging via JAX-RS ContainerRequestFilter

Status: Server is running and ready for testing.

Base URL

http://localhost:8080/api/v1
Use this base URL in Postman or with curl. All examples below assume Tomcat deployed the WAR as smart-campus-api. If your context path differs (e.g. smart-campus-api-1.0-SNAPSHOT), update accordingly.

1. Discovery Endpoint

GET/api/v1/

Request

GET http://localhost:8080/api/v1/

Expected Response

{
  "version": "1.0",
  "description": "Smart Campus Sensor & Room Management API",
  "contact": "admin@smartcampus.com",
  "resources": {
    "rooms": "/api/v1/rooms",
    "sensors": "/api/v1/sensors"
  }
}

2. Create Room

POST/api/v1/rooms

Header: Content-Type: application/json

Raw JSON Body

{
  "id": "R1",
  "name": "Lab 1",
  "capacity": 40,
  "sensorIds": []
}

curl (Windows)

curl -X POST http://localhost:8080/api/v1/rooms ^
-H "Content-Type: application/json" ^
-d "{\"id\":\"R1\",\"name\":\"Lab 1\",\"capacity\":40,\"sensorIds\":[]}"

Expected Response — 201 Created

{
  "id": "R1",
  "name": "Lab 1",
  "capacity": 40,
  "sensorIds": []
}

14. Delete Room with Sensors Inside (409 test)

R1 still has S1 and S2 attached. This should be blocked.

{

  "error": "Conflict",

Test Coverage Summary

  • 200 OK — successful GET requests and empty room DELETE
  • 201 Created — successful POST for room, sensor, and reading
  • 400 Bad Request — missing required field (id, name, roomId)
  • 403 Forbidden — posting a reading to a MAINTENANCE sensor
  • 404 Not Found — room or sensor does not exist
  • 409 Conflict — deleting a room that still has sensors
  • 415 Unsupported Media Type — wrong Content-Type header
  • 422 Unprocessable Entity — sensor created with non-existent roomId
  • 500 Internal Server Error — malformed JSON body
  • Filtering using ?type= query parameter
  • Sub-resource testing with /sensors/{id}/readings
